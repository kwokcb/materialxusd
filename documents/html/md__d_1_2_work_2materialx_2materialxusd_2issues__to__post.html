Improve Robustness of MaterialX to USD (USD components)

  • usdMtlx creates NodeGraphs for top level Nodes
    • UsdShade can support such configurations so appears to be an artificial constraint.
    • This complicates the graph and changes the original topology and exposure of nodes and inputs.
  • usdMtlx can fail if top level nodes not in NodeGraphs
    • Some nodes are not grouped under NodeGraphs and the import will fail. Seems PBR, BXDF nodes are not.
  • usdMtlx will create nothing if there is a connection failure
    • There is no concept of allow for valid parts to be imported.
  • usdMtlx imports nothing if there is no Material + shading model
    • This makes it impossible to import valid documents which for instance may only contain NodeGraphs and or Nodes
  • usdMtlx has no API to get errors back. This appears true for referencing or using Stage.Load() ?
  • Syntax for referencing MaterialX appears to not be documented. You need to specify the rootPrim as </MaterialX> otherwise "defaultPrim" is looked for. The error messaging is not very clear on this as well.
  • usdMtlx forms can form incorrect connections between NodeGraphs and downstream nodes.
    • This occurs if there are NodeGraph inputs with the same name as internal nodes. It appears to try to connect with the interior node and fails.
  • usdMtlx / UsdShade does not allow for BXDF / PBR nodes which have valid type surfaceshader, volumeshader, displacementshader outputs to be connected to downstream materials if they are not: glTF, USDPreview, std surface or OpenPBR.
    • This includes nodes which definition these shaders as their terminal node. This means related conversion nodes are not supported. e.g. "convert float to surface shader".
    • Documentation is not clear on what is allowed.
  • UsdShade requires output port specifiers for a valid connection. MaterialX does not always require this.
  • usdMtlx defaults to "out" as default upstream node / nodegraph output for connections
    • This may not be the case in which case connection creation fails and import aborts.
    • This is inconsistent with MaterialX which takes the first valid output.
  • usdMtlx requires explicit output qualifiers on MaterialX connections.
  • HDStorm seems to not considered defaultgeomprop on node definitions and hence requires explicit upstream stream node nodes to be connected.
  • usdMtlx "valid" graph configuratins is not well documented.
  • usdMtlx does not create a valid stage on Load. Could not find any documentation on this.
  • usdMtlx gives no indication as to what meta-data is not transferred. e.g. real world units which can greatly affect the computation is not transferred but no indication is provided.

Improve Robustness of MaterialX to USD Interoperability

  • MaterialX does not always require output qualifiers for connections. UsdShade does.
    • MaterialX considers explicit output qualifiers on upstream nodes/graphs which don't have multiple outputs to be invalid.
    • Should remove this restriction as invalid MaterialX is required for usdMtlx to create valid UsdShade graphs.
  • MaterialX nodes with single outputs fails validation if an output is specified.
  • Utility to convert implicit geometry bindings to explicit ones
  • Utility to add materials for terminal shading nodes.
  • Utility to add explicit output port qualifiers on input connections
  • Utility to add downstream shading nodes / materials for output connections.
  • Utility to group nodes into a nodegraph.
